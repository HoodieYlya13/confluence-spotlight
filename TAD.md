# Technical Architecture Decisions — confluence-spotlight

House convention: no code comments or docstrings. Every design decision and its rationale lives here.

## Purpose

A desktop **spotlight** client for the same [mcp-confluence-documentation-rag](https://github.com/HoodieYlya13/mcp-confluence-documentation-rag) server that the web console fronts. A global hotkey (`Cmd+Shift+Space` by default) summons a frameless, always-on-top command bar; you type an operations question, and the server's `ask_accelerator_operations` agent returns a short grounded answer with links straight to the source Confluence pages. It is the "ask in two seconds without leaving what you're doing" surface, where the web console is the "explore the RBAC story in a browser" surface.

This is a third client over one server — Python server, TypeScript web console, Rust desktop app — exercising the MCP protocol from a third language.

## Decisions

### The bearer token never leaves the Rust process — desktop `server-only`

This is the load-bearing decision and the direct analog of the web app's `server-only` token handling. The web console keeps tokens out of the browser; here the equivalent boundary is process-internal. The token is acquired by Rust (via the sign-in flow below), held in memory on the managed `AppState` (`auth: Mutex<Option<Auth>>`), and used only inside `ask_question`/`mcp::ask`. There is no command that returns a token. The webview is a thin renderer: it calls `ask_question` and receives back only `{ answer, role }` — rendered text and a cosmetic label. The token is never sent into the webview, never embedded in JS, never reachable from the page context.

The shipped build contains **no secrets at all**. `mcp.rs` (`McpConfig::from_env`, via `dotenvy`) reads only public configuration — `MCP_SERVER_URL`, the OIDC `SSO_ISSUER` and **public** `SSO_CLIENT_ID`, the demo `SPOTLIGHT_AUTH_URL`, the `SPOTLIGHT_USE_SSO` mode switch, and `SPOTLIGHT_HOTKEY` — all URLs/strings, with baked-in defaults so a distributed build works with no `.env`. No bearer token is compiled in: a release build obtains one from the OIDC provider at sign-in, a demo build from the console (the same `MCP_TOKEN_*` that power its playground), and it is held in memory only, never persisted. The login flow does not weaken the boundary: the webview triggers `begin_login` and later receives a success/failure event — it never names a token and never sees one. This mirrors the server's own model, where the client never sends its own role; identity is resolved from the bearer token everywhere in the system.

### A real Rust MCP client (`rmcp`), not a REST shim

`mcp.rs` uses the official Rust SDK `rmcp` with its streamable-HTTP client transport (`StreamableHttpClientTransport::from_config`), performing the full initialize → `call_tool("ask_accelerator_operations") → cancel` handshake per request. The bearer token is supplied through the transport config's `auth_header` (token without the `Bearer ` prefix; the transport adds it). A raw `reqwest` POST would have been fewer lines, but using the real SDK keeps this consistent with the web app's "real MCP client, not a REST shim" stance and means the project now demonstrates an MCP server plus clients in two different languages. Connections are per-request, matching the low-traffic, summon-on-demand usage.

The answer is extracted preferring the tool's `structured_content.result` (the string the agent returns) and falling back to concatenated text content — the same precedence the TypeScript client uses.

### Deep-link sign-in: real OIDC in production, a demo persona picker in dev — one Authorization-Code + PKCE shape

The spotlight runs as **one** role at a time (a personal tool, not the console's side-by-side RBAC diff). It is a *public* client — a desktop binary cannot hold a client secret — so sign-in is always Authorization-Code **with PKCE**, and the bearer is redeemed by **Rust, never the webview**. The same outer flow drives two interchangeable back ends, selected by `McpConfig.use_sso` (`SPOTLIGHT_USE_SSO`, defaulting to `!debug_assertions` — real SSO in a release build, the demo path under `tauri dev`).

The shared shape:

1. **Connect** invokes `begin_login` (Rust): generate `state` (CSRF) + a PKCE `verifier`, derive `challenge = base64url(sha256(verifier))`, store `{state, verifier, redirect_uri}` in `pending_auth`, and open the system browser (`tauri-plugin-opener`). The `redirect_uri` is read from the app's **own** config (`deep_link_redirect_uri`), so the release and Beta builds each send their matching scheme. Once the browser is open the webview hides the bar (`hide_window`): sign-in has handed off to the browser, so the spotlight gets out of the way rather than sitting behind it on a "waiting" screen.
2. The browser authenticates the user and hands an authorization `code` back over the custom scheme; `tauri-plugin-deep-link`'s `on_open_url` fires in the running process. The arrival of that redirect is what brings the bar back: `handle_auth_url` re-shows the window as soon as a `code`+`state` redirect lands (before the async token exchange), so the bar reappears on *both* success and failure and the result is always visible. Rust then checks the returned `state` against `pending_auth`, exchanges the `code` for a token, stores it in `auth`, and emits `spotlight-auth` so the webview switches to the search view (or renders the error). The show lives at the redirect boundary, not in `finish_login`, so the error paths surface in a visible window too.

**Production — real OpenID Connect (`use_sso`).** `begin_login` builds an `/api/oidc/authorize` URL against the self-hosted issuer (`SSO_ISSUER`, default `auth.hy13dev.com`) with `response_type=code`, the public `SSO_CLIENT_ID`, `code_challenge_method=S256`, a `nonce`, and `scope=openid profile email offline_access` (the `offline_access` is what yields a refresh token). The user authenticates against the IdP — there is **no role picker in the app**. On return, `sso_exchange_code` POSTs `grant_type=authorization_code` to `/api/oidc/token` and receives `{access_token, refresh_token, expires_in}`. The **role is read from the signed access token, not chosen**: `role_from_access_token` decodes the JWT's `roles` claim and maps it (`ADMIN`/`ATS_CORE_LEAD` → `ATS_CORE_LEAD`, `JUNIOR_OP` → `JUNIOR_OP`), and **fails closed** — an account with no recognized role is refused ("no authorized role for this app") rather than admitted role-less. `expires_in` becomes an absolute `expires_at` (minus a 30 s skew); `refresh_if_needed`, called before every `ask`, silently rotates an expired access token via `sso_refresh` (`grant_type=refresh_token`), carrying the rotated refresh token forward.

**Dev / demo — the console persona picker (`!use_sso`).** `begin_login` instead opens the console's `/spotlight-login?state=…&code_challenge=…&redirect_uri=…`. That page (the login surface — role selection belongs there, not in the app) validates the `redirect_uri` against its allowlist and renders one persona button per configured role; picking one runs a server action that mints a single-use authorization `code` (60 s TTL in Upstash, keyed to role + `challenge` + `state`) and redirects back. `exchange_code` POSTs the `code` + `verifier` to `/api/spotlight/token`, which re-verifies PKCE S256 and returns a *pre-provisioned demo token* plus its `role`/`role_label`. The in-app **Dev sign-in** shortcut (below) is a no-browser stand-in for this same path.

**Honest framing.** The demo path's token endpoint vends a pre-provisioned, role-fixed token — it stands in for an IdP's token endpoint rather than minting a per-user credential. The SSO path is the genuine article end to end: a real OIDC provider issues per-account access + refresh tokens, the clearance lives in the signed token's claims, and the MCP server verifies that token against the provider's JWKS (issuer, audience, RS256) before resolving its role. The server accepts **both** bearer kinds against one RBAC model, so the demo and real federated identity are interchangeable.

**Tokens are ephemeral and never written to disk.** `auth` (the access token, and for SSO the refresh token + `expires_at`) lives only in memory on the managed `AppState`. Only the custom hotkey is persisted, to `session.json` (`app_config_dir()`, written by `save_settings`) — a single accelerator string, no role and no token. So the app reconnects on every launch; within a session the SSO refresh token renews an expired access token in place. (Persisting the refresh token in the OS keychain to survive restarts is the remaining deferred step.)

**Identity, surfaced from the token's claims — not just the role.** The role is still read from the signed access token for RBAC (`role_from_access_token` → `roles` claim), but the UI now speaks in terms of the *person*, not the clearance. `mcp::profile_from_access_token` decodes the same JWT for `preferred_username`, the optional `given_name`, and `iss`, carrying them on `Auth` and out through `SessionView`. The search bar no longer shows a role pill at all — instead the input placeholder greets the user (`Hello {given_name ?? preferred_username} ! Ask the accelerator operations assistant…`), keeping the chrome about identity rather than label. A gear opens **Settings**, whose **Identity** row shows the `preferred_username` (falling back to the role label when a demo/opaque token carries no profile claims) next to a **Manage account** button that opens `{iss}/en/account` in the system browser (`tauri-plugin-opener`), and a **Log out** action that clears `auth` and returns to the connect screen — in a dev build to switch persona, in a release build to sign in as a different account. Because the claims are optional, the whole chain degrades cleanly: a token with no `preferred_username`/`iss` (the demo path) simply shows the role label and hides the account button.

### Customizable global hotkey, validated by re-registration

Settings lets the user record a new summon shortcut: the webview captures one `keydown`, builds a Tauri accelerator string from the modifiers + `event.code` (`toAccelerator`), and requires at least one non-Shift modifier so a bare key cannot be captured. The candidate is only persisted on explicit Save. Validation is not duplicated in the frontend — `set_hotkey` unregisters the current shortcut, attempts to register the candidate, and on failure re-registers the old one and returns the error to the UI. So the global-shortcut plugin's own parser is the single source of truth for what is a valid accelerator; an unparseable or unavailable combination surfaces as an inline error and nothing is saved. The accepted hotkey is the only thing persisted to `session.json` and is re-registered on next launch.

### Cancelling a slow request (`Ctrl+C`)

The agent call can take a while — a cold free-tier Space waking up, a long generation — so while a question is in flight the status line shows a right-aligned `⌃C to cancel` hint and `Ctrl+C` aborts it. Cancellation is **real, not cosmetic**: `ask_question` runs the `mcp::ask` future inside a `tokio::select!` against an `Arc<Notify>` held on `AppState`, and the `cancel_ask` command calls `notify_waiters()`. Losing the race drops the `mcp::ask` future, which drops the in-flight `rmcp`/`reqwest` request, so the HTTP connection is actually torn down rather than left running server-bound. The command returns a `__cancelled__` sentinel the webview renders as a quiet "Request cancelled." instead of an error. `Ctrl+C` (not `⌘C`) is used so it never shadows copy on macOS, and the binding is only live while a request is pending — otherwise the keystroke passes through untouched.

### In-window keyboard navigation: history, scrolling, numbered links

The bar is keyboard-first, so beyond the global summon hotkey there is a set of **in-window** shortcuts handled entirely in the webview (`main.ts` `keydown`/`keyup`), not via the OS global-shortcut plugin — they only matter while the window has focus, so there is nothing to register with the system and nothing to leak between apps.

- **Question history (`↑`/`↓`, fixed).** The search box behaves like a shell prompt: `↑` walks back through questions asked in the **last 24 hours**, `↓` walks forward, and stepping past the newest restores the in-progress draft. Pressing `↓` *again* once you're back at that bottom draft erases it — a second `↓` at the bottom empties the line — but the erase is reversible: the draft is held in `erasedDraft`, and the next `↑` restores it before stepping into history (so down-erases, up-brings-it-back). Typing a new draft or submitting discards the held one. An empty bottom (nothing ever typed) has nothing to erase, so `↓` there is a no-op and `↑` goes straight to history. History is **in-memory only** — a timestamped array pruned to 24 h on access, never written to disk. Questions aren't secrets, but keeping them out of `session.json` holds the line that the only thing persisted is configuration (the hotkey + the remappable bindings below). Plain `↑`/`↓` is the terminal convention and is deliberately *not* remappable, the way no shell lets you rebind up-for-history.
- **Reset on submit.** Submitting clears the input (and pushes it to history). This fixes a real bug — previously a second Enter re-asked the same question — and makes history the way you recall a prior query rather than retyping.
- **Scroll the answer (`⌘/Ctrl+↑/↓`).** Pages the answer body without reaching for the trackpad. The same shortcut scrolls the **Settings** screen when it is open (and you are not mid-remap), so long settings stay keyboard-reachable.
- **Jump to the sources (hold `⌘/Ctrl+Shift`, repeatable with `↓`).** Because the global summon is `⌘/Ctrl+Shift+Space`, the bare-chord trigger is **debounced ~300 ms**: holding `⌘/Ctrl+Shift` starts a timer that is cancelled the moment `Space` (or any other key) follows, so reaching for the summon never accidentally scrolls. If the chord is still held alone after the delay, the view scrolls so the **first** source link sits at the top — never past it, which matters when an answer has many links — and enters *link mode*, numbering every link `0,1,2,…` with an inline badge. `⌘/Ctrl+Shift+↓`/`↑` then page through any overflow. (The arrow form fires immediately, cancelling the timer, so `⌘/Ctrl+Shift+↓` is still the explicit, no-wait way in.)
- **Open a link by number (`⌘/Ctrl+(Shift)+<digit>`).** Digits build a buffer matched as a **decimal prefix** of the link indices, so the logic scales to any link count:
  - no index starts with the buffer → reset (e.g. `13` with 12 links);
  - exactly one index matches and nothing longer can extend it → open immediately (e.g. `3` when there is no `30+`, or any single digit when there are ≤ 10 links);
  - several still match → hide the non-matching candidates so the list visibly collapses to just `1, 10–19, 100–199…`, and wait for the next digit.

  **Only numeric badges are filtered.** `applyLinkFilter` hides a target only when its badge is a number that no longer prefix-matches the buffer; letter badges (`U` for the update banner, `J`/`L`/`⏎` on the login screen) are always eligible, so typing a digit never hides them. Hiding uses `.link-hide { display: none !important }` — the `!important` is required because the per-element display rules (`.answer .link` is `inline-block`, `.setting-row` is `flex`) have equal-or-higher specificity and would otherwise win, leaving nothing actually hidden.
  The ambiguous short case (you want link `1` while `10–19` exist) resolves on **key release**: while the base modifier stays held, each digit extends the buffer; releasing it commits the current buffer if it is a valid index. So holding-and-typing reaches the long numbers and a quick press-release opens the short one — and when there is no ambiguity it never waits for the release at all.
  - **Shift is the entry modifier, not the activation modifier.** `Shift` is required only to *engage* link mode (`linkChordEngaged` keeps the full `⌘/Ctrl+Shift` so the trigger stays distinct from the `⌘/Ctrl+↓` scroll chord). Once numbers are on screen, link mode is bound to the **base modifier alone** (`linkChordHeldNow` → `baseModsHeld`, which ignores `Shift`): digits activate with `⌘/Ctrl+<digit>` whether or not `Shift` is held, and only releasing `⌘/Ctrl` ends link mode / commits the buffer. This is deliberate — **macOS reserves `⌘⇧3/4/5/6` for screenshots at the WindowServer level, so those keydowns never reach the webview**; making `Shift` optional lets the user release it and press `⌘+3/4/5/6` to reach those links. The digits `3/4/5/6` are not special-cased in code — `⌘⇧3/4/5/6` is still accepted and simply never arrives on macOS, which keeps the handler uniform. (On Windows/Linux there is no such conflict and the full chord works for every digit.)
  A fresh hold re-enters link mode and scrolls to the sources again, so the numbering is only ever on screen while you are actually using it. `Esc` also exits link mode (clearing the buffer) before it falls through to hiding the window.
- **Settings navigation.** `⌘/Ctrl+,` toggles the Settings screen open and closed; from Settings, `Esc`, `←`, or `⌘/Ctrl+,` return to the conversation, and `Shift+Q` logs out — pressed **twice** within ~2.5 s to confirm. The first press arms the action (the Log out button relabels to "Press ⇧Q again to log out") and a timer disarms it, so a single stray `⇧Q` can't sign you out; leaving Settings disarms it too.
- **Numbered buttons in Settings.** The link chord is view-aware: holding it over the Settings screen (`⌘/Ctrl+Shift`, or `Space+Shift` in Neovim mode) numbers the actionable buttons — `Log out` = 0 through `Check for updates` — and the same prefix/release logic activates one. `computeTargets()` returns the answer links in the conversation and the visible Settings buttons (DOM order, minus the Back button) in Settings; the number badge is a CSS `::before` on the target so it sits inside links and buttons without disturbing layout, and `activateTarget` opens a link's URL or `.click()`s a button. Filtering in Settings collapses at the **row** level: `hideScopeFor` resolves each numbered button to its enclosing `.setting-row`, and a row is hidden only when *all* of its numbered buttons are ineligible — so a multi-button row (e.g. **Open in** → Insert/Normal) stays visible and hides just the non-matching button, while single-button rows collapse entirely to make the matching options easier to scan.

**The input keeps focus.** In the search view the caret stays in the search box whenever you click around the bar — chrome clicks `preventDefault` the focus shift, and a click in the answer body refocuses the input unless you are selecting text — so you can start typing the next question at any moment without clicking back into the field.

**Remappable, like the global hotkey.** The three modifier-based families — scroll, jump-to-links, open-settings (`⌘/Ctrl+,`) — are recorded with the same capture-one-keystroke recorder as the global hotkey and persisted to `session.json` via `set_binding` (`scroll_keys` / `link_keys` / `settings_keys`, each defaulting in `session_view` when unset). Matching is modifier-aware (`CmdOrCtrl` accepts either ⌘ or Ctrl); the two directional families apply their recorded modifiers to both arrows, and the digit-to-open shortcut reuses the link family's modifiers, so rebinding "jump to links" moves the number shortcuts with it. These bindings are window-local config, so unlike the global hotkey they need no OS (un)registration — saving is a plain persist.

### Neovim mode: an optional modal keymap layered over the bar

An off-by-default **Neovim mode** toggle (persisted in `session.json` alongside the bindings) adds a vim-style modal layer for users who'd rather drive everything from the home row. It's purely additive: the base `⌘/Ctrl` shortcuts keep working unchanged in both modes, and with the toggle off the bar behaves exactly as before.

**The input never loses focus; the mode decides what keys mean.** This is the load-bearing idea — the `#question` field keeps DOM focus in both modes (the same "type at any moment" guarantee as non-vim mode), and `handleNvim` in the capture-phase `keydown` decides per keystroke whether to let the character type (Insert) or interpret it as a command (Normal), `preventDefault`-ing the latter. A `NORMAL`/`INSERT`/`VISUAL` badge in the bar names the current mode. The **cursor shape matches Neovim too**: a native `<input>` can't style its caret into a block, so Normal mode fakes one — `renderCursorAt` paints the cursor as a *one-character forward selection* (`setSelectionRange(c, c+1)`) under a `.nvim-block` class that styles the selected glyph as a solid block (`::selection` → accent background, bg-colored text), while a mode-level `.nvim-active` class hides the real caret (`caret-color: transparent`) for the *whole* of Normal/Visual — including the empty field, where there's no glyph to select and a small absolutely-positioned `#nvim-cursor` overlay stands in for the block (the earlier "empty field shows a native bar" bug). Insert mode drops both classes and restores the plain collapsed caret. This reuses the existing `selectionStart`-as-caret model, so every motion that already moved the caret now moves the block for free; the block clamps to the last real character (no sitting one past the end), leaving Esc its Neovim "shift left by one" behaviour. **Visual mode** is the same idea widened: the native selection *is* the visual range (a `.nvim-visual` class tints it differently from the one-char block), `visualAnchor`/`visualCursor` track the ends, and *any* selection — mouse drag **or** keyboard (Shift-arrows in Insert) — auto-enters it via a document `selectionchange` listener. That event fires asynchronously, so a synchronous "is this programmatic" flag can't gate it; instead `setSelectionSilently` records the range it just set in `lastSetSelection` and the handler ignores an event whose range matches, so the cursor's own one-char/visual selections don't re-trigger entry. It also ignores any event where `input.value` changed since the last one (`lastSelValue`): a selection that shifts *because the text changed* — history navigation reassigning `input.value`, typing — is not a selection gesture, so only mouse drags / Shift-arrows / `⌘A` (which move the selection without changing the text) enter Visual. Without that guard, history `↑`/`↓` flipped into Visual mid-navigation and the arrows stopped paging. Reacting to the *committed* selection rather than the mouse position is what makes it reliable regardless of where the drag ends. An **Open in** setting (Insert by default) picks the mode each summon — die-hard vim users still usually want to start typing immediately.

- **Insert → Normal:** `Esc`, `Ctrl+C`, `Ctrl+[`, or `jj` within 250 ms (the second `j` is deleted and the mode flips), and automatically after every `Enter` (submit). `Ctrl+C` cancels an in-flight request first if one is pending, otherwise it switches mode.
- **Normal → Insert:** `i`/`a`/`I`/`A` with the correct caret (`i` at caret, `a` after it, `I` at start, `A` at end).
- **Normal motions:** `h`/`l` move the cursor (clamped to the last real character, like Neovim — you can't sit one past the end), `Backspace` moves left like `h` (it's `preventDefault`-ed, so it never deletes or collapses the block — matching Neovim, where `<BS>` is a motion in Normal mode), `k`/`j` step history (older/newer), `0`/`$` jump to line start/end. Any other bare key is swallowed so nothing leaks into the field.
- **Normal edits & registers:** `x` deletes the char under the cursor, `dd` (double-tap within 250 ms) deletes the whole line, `D` deletes to end of line, `y`/`yy`/`Y` yank the line, `p`/`P` paste after/before the cursor, `u` undoes and `Ctrl+R` redoes. Yank/delete fill an internal register *and* push to the system clipboard, and paste reads the **system clipboard first** (so a copy from any other app pastes here), falling back to the internal register only if the read fails. Clipboard I/O goes through `tauri-plugin-clipboard-manager` (Rust-side read/write), **not** `navigator.clipboard` — the webview's `readText()` triggers a macOS permission prompt on every paste, whereas the plugin reads the native pasteboard silently. undo/redo are explicit value+cursor snapshot stacks captured before each mutating command and on entering Insert, so an Insert session collapses to a single undo step.
- **Visual mode:** `v` enters it from Normal (and any selection — mouse drag or keyboard Shift-arrows — auto-enters it); `h`/`l`/`0`/`$` extend the selection, `y` yanks it, `d`/`x` delete it, `c` deletes and drops into Insert, `Esc`/`v` collapse back to Normal (and `Esc` primes the double-`Esc`-to-hide timer, so two Escs from Visual close the panel just like from Normal). The selection is the native one (inclusive of the char under the cursor), so it survives the round trip to the register and the clipboard.
- **Leader is Space, used as a held modifier** (not a sequential prefix) — this is what makes `<leader>+Shift` able to mirror the real `⌘/Ctrl+Shift` link chord *with release semantics*: `linkChordEngaged` treats "`Space` held + `Shift`" exactly like the physical chord to engage, and `linkChordHeldNow` then holds link mode on `Space` alone (Shift optional, mirroring the base-modifier rule above). So holding `<leader>+Shift` scrolls-and-numbers after the same 300 ms debounce, `<leader>+<digit>` opens with the same prefix logic whether or not `Shift` is held, and releasing `Space` commits — no separate code path, just a second way to satisfy the chord predicate. `<leader>j`/`k` scroll the answer, `<leader>,` toggles Settings, `<leader>q` (or a quick `qq`) closes the window.
- **In Settings (Normal):** `h`, `Backspace`, `Esc` (or the Insert→Normal keys) return to the conversation, `j`/`k` scroll the settings body, and `<leader>Shift+Q` arms logout with the same press-`q`-again-within-2.5 s confirmation as the button.

**Remap scope.** Only the **leader key** and the **Enter-Normal key** are remappable (recorded with a single-key variant of the recorder that accepts bare keys like `Space`/`Esc`, stored as a key `code`); the motions themselves are fixed vim conventions, the same call made for the base directional shortcuts. `Ctrl+C`/`Ctrl+[`/`jj` remain always-on Normal triggers regardless of the remapped key. Known trade-off, inherent to the `jj` escape: you can't quickly type a literal "jj" in Insert mode.

### Frameless NSPanel that floats over fullscreen apps, dismiss-on-blur

`tauri.conf.json` starts the window hidden (`visible: false`) and frameless (`decorations: false`, `transparent: true`, `skipTaskbar: true`, `resizable: false`, `center: true`). The global shortcut (registered in Rust via `tauri-plugin-global-shortcut`, `on_shortcut`) toggles the panel, centers it, and emits `spotlight-open` so the webview focuses and selects the input. Pressing Esc (a `hide_window` command) hides it. Transparency requires `macOSPrivateApi: true` plus the `macos-private-api` cargo feature on the `tauri` dependency.

A plain always-on-top window only floats over *ordinary* windows: macOS gives each fullscreen app its own Space, and a regular window cannot appear in it — and even if forced to, activating a regular app to give it keyboard focus yanks the user out of the fullscreen Space. The standard fix, and what every Raycast/Spotlight-style app uses, is an `NSPanel`. So in `setup` the window is converted with `tauri-nspanel`'s `to_panel` (the `tauri_panel!` macro declares the panel class as `can_become_key_window` + `is_floating_panel`), then configured for the overlay contract:

- `set_collection_behavior(full_screen_auxiliary | can_join_all_spaces)` — the panel is allowed *into* the active fullscreen Space and onto every Space, instead of being confined to its own.
- `set_style_mask(nonactivating_panel)` — the panel can become key and take keystrokes **without activating the app**, so summoning it never switches Spaces or steals focus from the app underneath.
- `set_level(Floating)` — sits above normal window content.
- `set_activation_policy(Accessory)` — no Dock icon and no Cmd-Tab entry, matching a summon-on-hotkey utility. (Trade-off: a packaged build has no Dock icon to quit from; under `tauri dev` the terminal owns the process. Removing this one line restores the Dock icon.)

Dismiss-on-blur is handled by the panel's own `window_did_resign_key` delegate (registered via the macro's `panel_event!` and `set_event_handler`) rather than Tauri's `WindowEvent::Focused`, which a non-activating panel does not emit reliably.

**The bar always opens on launch.** The window is declared `visible: false` so nothing flashes before the panel is installed and positioned, and `setup` then *unconditionally* calls `show_window` on the main thread — so the bar is open and key on every start: boot/login-item, a manual open, a quit-and-reopen, and the post-update restart. After the user dismisses it (blur or `Esc`), the global hotkey re-summons. This replaced an earlier `show_on_launch` flag that `install_update` set just before restarting: it only addressed the update case and proved unreliable across the relaunch, where "always show on launch" is both simpler and what the app actually wants — the first thing after any launch is the user wanting to ask something.

Because the window is `transparent`, the OS clips everything to the window bounds — including the card's CSS drop shadow. The card therefore does not fill the window: the `body` reserves padding (`36px 48px 72px`, more at the bottom to match the shadow's downward offset) and the card is height-bounded to `calc(100vh - 108px)`, so the `0 18px 48px` shadow fades out inside the transparent window instead of being cut off at the edges.

**Cross-platform fallback.** The `NSPanel` machinery is macOS-only, so `tauri-nspanel` (and everything that touches it — the `tauri_panel!` macro, `to_panel`, `set_collection_behavior`, `set_style_mask`, `set_activation_policy`, the panel show/hide/toggle paths) lives behind `#[cfg(target_os = "macos")]`, and the crate is declared under `[target.'cfg(target_os = "macos")'.dependencies]` so it never compiles on Windows/Linux. There, `show_window`/`hide_window_impl`/`toggle_window` operate on the ordinary `WebviewWindow` (still frameless, `transparent`, `alwaysOnTop`, `skipTaskbar`), and dismiss-on-blur is wired through Tauri's `WindowEvent::Focused(false)` — which a normal window *does* emit, unlike the non-activating panel. The window contract is otherwise identical, so one `tauri.conf.json` drives all three platforms; only the focus/overlay mechanism is conditional. (`macos-private-api` stays a plain feature on the base `tauri` dependency — it is inert off macOS, and keeping it in `[dependencies]` satisfies `tauri-build`'s check that the feature matches `macOSPrivateApi: true` in the config.)

### Links open in the system browser, not the webview

Confluence pages must open in the user's real browser, not the tiny always-on-top webview. The Markdown renderer emits `<a data-href="…">`, and a delegated click handler in `main.ts` calls `tauri-plugin-opener`'s `openUrl`. The capability set therefore adds `opener:allow-open-url`.

Source links render on a **single line** — `inline-block` with `white-space: nowrap` and `text-overflow: ellipsis` — so a long title or URL truncates instead of wrapping onto a second line. In the cramped overlay this keeps the source list scannable and, with link mode active, keeps each inline number aligned to exactly one link row.

### Vanilla TS + a ported Markdown renderer

No React/Tailwind: the hover window must be tiny and instant, so the frontend is vanilla TS + Vite. `markdown.ts` is a dependency-free port of the web app's `parseMarkdown` (headings, lists, tables, code, and the inline pass for `**bold**`, `` `code` ``, `[text](url)`, `0x…` hex). It differs in two ways: it emits **escaped** HTML strings (the agent answer is treated as untrusted input, consistent with the server's prompt-injection stance — escape first, then re-introduce only the known constructs), and the `0x…` hex highlight is purely cosmetic here because there is no second role to diff against.

### No client-side rate limiting

The spotlight is a single-user desktop tool, so it has no limiter of its own; the server remains the shared, protected resource. The two console endpoints it now depends on (the authorize action and the token exchange) *are* fronted by the console's existing Upstash limiter, so distributing the app does not open an unmetered path to the bearer tokens.

### Custom URL scheme and the macOS development caveat

The deep link uses a private scheme, `confluence-spotlight://auth`, declared under `plugins.deep-link.desktop.schemes` in `tauri.conf.json` and registered with LaunchServices via the bundle's generated `Info.plist`. **On macOS this means the round trip only works in a bundled app installed in `/Applications` — not under `tauri dev`** (the deep-link plugin cannot register a scheme at runtime on macOS). So the full sign-in is verified on the installed build; the scheme handler itself can be exercised in isolation with `open "confluence-spotlight://auth?code=…&state=…"`. `register_all()` is called only on Windows/Linux (where runtime registration *is* supported) to keep `tauri dev` working there. The redirect URI the app sends is derived from its **own** config at runtime — `deep_link_redirect_uri` reads `plugins.deep-link.desktop.schemes` from the embedded `tauri.conf.json` — so a build configured with a different scheme (the Beta build, below) sends a matching `redirect_uri` with no code change, and the console allowlists both.

Since that round trip dead-ends under `tauri dev`, a **dev-only sign-in fallback** keeps the app usable while iterating: the `dev_login` command reads `MCP_TOKEN_<role>` from `.env` and sets `auth` directly, and the login view exposes one persona button per role. Both halves are gated so the production posture is unchanged — the command body and its `dev_token` helper are compiled only under `#[cfg(debug_assertions)]` (a release build's `dev_login` is an inert `Err`, with no token-reading code in the binary), and the webview reveals the buttons only when `import.meta.env.DEV` (false in a `vite build`). The fallback deliberately bypasses the browser/PKCE exchange — it is a dev shortcut to a token, not a second auth path — so the shipped build still has exactly one way in.

## Deployment

Releases are cut by a GitHub Actions matrix (`.github/workflows/release.yml`, `tauri-apps/tauri-action`) that builds on `macos-latest` (universal `aarch64`+`x86_64` `.dmg`), `ubuntu-22.04` (`.AppImage`/`.deb`), and `windows-latest` (`.exe`/`.msi`) on every `v*` tag, attaching every artifact to a single GitHub release — the "downloadable on every OS" story.

**One release, created up front.** The matrix runs in parallel, so the release must *not* be created from inside it: if each platform job passed `tagName` and "create the release if it's missing", the three jobs race and two of them each create a separate release for the same tag, scattering the artifacts (and `latest.json`) across duplicates — the bug that produced two `v0.1.11` releases. Instead the workflow is four phases: a `cleanup` job, a `create-release` job that creates the draft **once** and outputs its `release_id`, the matrix `release` job which passes that `releaseId` (never `tagName`) so all three platforms upload into the *same* release, and a final `publish-release` job that flips the draft to published only after every build has uploaded. A half-built tag therefore never appears as "Latest". The builds are intentionally **unsigned/un-notarized** (this is a demo), so first launch needs the per-OS bypass (Gatekeeper right-click → Open or `xattr -dr com.apple.quarantine`; SmartScreen → Run anyway; `chmod +x` the AppImage). Because the shipped binary holds no secrets, build-time configuration is only public values, all in `mcp.rs`: `DEFAULT_SERVER_URL` (the MCP server) and — for the default-on SSO path of a release build — `DEFAULT_SSO_ISSUER` plus the **public** `DEFAULT_SSO_CLIENT_ID`; `DEFAULT_AUTH_URL` is the console, used only by the dev/demo sign-in. Each is overridable at build time (the `option_env!` constants) or at runtime (`MCP_SERVER_URL`, `SSO_ISSUER`, `SSO_CLIENT_ID`, `SPOTLIGHT_AUTH_URL`), and `SPOTLIGHT_USE_SSO` can force the mode. A release build therefore needs `SSO_ISSUER` + `SSO_CLIENT_ID` pointed at the deployed identity provider; the demo build needs `SPOTLIGHT_AUTH_URL` pointed at the console.

**Re-tagging is idempotent on purpose.** Building the *same* tag twice (a re-pushed or moved tag, or a `workflow_dispatch` on top of the tag push) is the failure that corrupts an updater release: `tauri-action` merges into the existing draft's `latest.json`, and a second run can leave the uploaded macOS tarball signed by run B while `latest.json` still carries run A's signature — every client then fails `minisign` verification with "signature verification failed" even though the key is correct. Two guards make a rebuild safe: a workflow-level `concurrency` group (`cancel-in-progress`) kills an in-flight run when a newer one starts, and the `cleanup` job (which `create-release` depends on) deletes **every** release whose `tag_name` matches — drafts included — before the new one is created. Deleting via the REST API by id rather than `gh release delete <tag>` matters: a leftover draft has no published tag ref, so the by-tag command silently skips it, and stale drafts would otherwise accumulate across re-runs. Net effect: you can re-push a tag to ship a quick fix without cutting a new version, and the release can never end up half-old or duplicated.

### A "Beta" build that coexists with the release

`bun run tauri:beta` builds with a config overlay (`tauri.beta.conf.json`, merged over the base via `tauri build --config`) that changes exactly three things: the product name (**Confluence Spotlight Beta**), the bundle identifier (`…confluence-spotlight-beta`), and the deep-link scheme (`confluence-spotlight-beta://`). The distinct identifier gives Beta its own install slot and its own `session.json`; the distinct scheme means its sign-in deep link routes to *it* rather than to an installed release — so a locally-built Beta and the downloaded release can be installed and used at the same time without LaunchServices ambiguity. This works only because the redirect URI is read from config (above) and the console allowlists both schemes, so no Rust or console code branches on "beta". The base `tauri.conf.json` stays the canonical published identity, so CI releases are unaffected.

### In-app self-update over the GitHub releases feed

A distributed binary has no package manager behind it, so the app updates itself with Tauri's updater plugin (`tauri-plugin-updater`) rather than asking users to revisit the download page. The base `tauri.conf.json` configures one static endpoint — `…/releases/latest/download/latest.json` — and a `pubkey`; `bundle.createUpdaterArtifacts: true` makes `tauri-action` emit the per-platform updater bundles (`.app.tar.gz`, NSIS, `.AppImage`) with `.sig` files and aggregate them into the `latest.json` manifest it uploads to each release.

**Signature verification is mandatory and the trust root is baked in.** The plugin will not install an artifact whose minisign signature doesn't verify against the embedded `pubkey`, so the update channel can't be hijacked by a tampered release asset or a compromised mirror — the private key never leaves CI (GitHub secrets `TAURI_SIGNING_PRIVATE_KEY` / `…_PASSWORD`), the public key ships in the app. This is the update-time analog of the rest of the app's posture: the client trusts only what it can verify. (Consequence: a release `tauri build` requires the signing key in the environment, by design — an unsigned "release" would produce updates nobody could install.)

**The check is driven from Rust, not the webview**, matching where the rest of the app's privileged logic lives (`begin_login`, the token exchange). On launch a background task calls `app.updater().check()`; if a newer version is published it emits `spotlight-update-available` `{version, current_version, notes}`, and the webview shows a dismissible **Update & restart** banner. The webview affordances are two thin commands — `install_update` (re-checks, `download_and_install`s, `app.restart()`s) and `check_update` (returns `{available, version, notes}` for the **Settings → Updates → Check for updates** button) — so the page never gets an updater handle of its own and no extra capability is exposed. The manual `check_update` is deliberately *not* gated on `not(debug_assertions)` like the launch check: it is user-initiated, so it runs in any build (in dev it surfaces the pubkey-not-configured error, which is a useful wiring signal), and on success it reuses the same banner the auto-check raises.

**Gated to where it makes sense.** The check is `#[cfg(all(desktop, not(debug_assertions)))]` — it never runs under `tauri dev` (no signed feed; the version would equal the running one) — and it is skipped when the bundle identifier ends in `-beta`, because the Beta build is a *local* build with no release feed of its own and would otherwise be offered the release installer (a different identity). The Beta overlay also flips `createUpdaterArtifacts` back to `false`, so `bun run tauri:beta` needs no signing key and produces no updater artifacts. Shipping an update is therefore: bump the version in the three manifests, tag, let CI sign and draft, then **publish** the release — only a published, non-pre-release tag resolves through `releases/latest`.

## Verification

- **Console half (no GUI):** drive `/spotlight-login`'s authorize action to mint a `code`, then `POST /api/spotlight/token` with the matching PKCE `code_verifier` — asserts a valid exchange returns the token and that replay (single-use), wrong verifier, and expired/unknown codes are all rejected.
- **Core data path:** `cargo run --example probe` (with `MCP_TOKEN_JUNIOR_OP`/`MCP_SERVER_URL` set; `SPOTLIGHT_ROLE` selects which role's token, default `JUNIOR_OP`) calls `mcp::ask` against the live server and prints the grounded answer + source links — the GUI-free proof the client works.
- **Compiles:** `cargo check --examples`, `cargo clippy --examples`, `cargo fmt --check`; frontend `bun run build` (runs `tsc`).
- **Manual (installed build, macOS):** install the `.app` to `/Applications`, summon, **Connect** → authenticate in the browser (the IdP in a release build, the console's persona picker in a demo build) → approve the "Open Confluence Spotlight?" prompt → returns signed in → ask, click a source link, Esc/blur to dismiss, gear → Log out to sign in again.
- **Self-update:** with the signing keys wired and the pubkey set, install an older release, publish a higher-versioned signed release, relaunch — the **Update & restart** banner should appear (or trigger it from **Settings → Updates**), and accepting it should download, verify, install, and relaunch into the new version.
